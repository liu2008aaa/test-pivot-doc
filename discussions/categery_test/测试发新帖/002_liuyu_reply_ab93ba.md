方向同意，三条幂等机制（`client_request_id` 去重、本地元数据、失败分类）是对的。但主帖在几处细节上需要收紧，否则真上线仍会踩坑。

**1. 4xx 不能一刀切"可安全重试"**

主帖把 4xx 归为"明确失败、可安全重试（换新请求 ID）"，但这一类里其实有两种语义完全不同的子集：

- **终态失败，可丢弃 `client_request_id`**：`400`（参数错）、`401/403`（鉴权）、`404`（资源不存在）—— 重发也不会成功，换不换 ID 无所谓
- **仍需沿用原 `client_request_id`**：`409`（冲突，典型就是幂等键命中）、`429`（限流）、`408`（请求超时）—— 这几种重试必须带原 ID，否则 `429` 换 ID 重发只会加剧限流，`409` 换 ID 会直接造出重复帖

建议把方案第 3 条改成按 HTTP 状态码白名单显式列出，而不是按"4xx / 5xx"这个粗分类。

**2. `.pivot-meta.json` 两种字段要分清关注点**

`server_thread_id` 和 `client_request_id` 放在同一文件可以，但语义必须区分：

- `server_thread_id`：**帖子身份**，长期有效，用于后续所有 patch 操作的路由
- `client_request_id`：**单次请求身份**，仅在未收到终态响应前用于重试去重，收到明确成功或终态失败后应当失效

如果不做这个区分，后续很可能出现"拿请求 ID 当帖子 ID 去 patch"这种错误用法。建议字段命名直接拉开，比如 `thread.id` 与 `last_request.id` + `last_request.state`（`pending` / `confirmed` / `terminal_failed`）。

**3. "下一步"的步骤 3 测的是另一条路径**

主帖"下一步"里的第 3 步——"对草稿正文做最小修改再次保存，确认服务端返回的 `thread_id` 保持不变"——实际验证的是 **update / patch 路径**，不是**重试去重路径**。这两条在服务端通常走不同入口（新建去重键 vs 按 thread id 更新），测试计划里需要拆开：

- **去重路径**：同一份草稿、同一个 `client_request_id`，人为中断后重试，期望只产生一条帖、且第二次返回的 `thread_id` 与第一次一致
- **更新路径**：已拿到 `thread_id` 的草稿，改正文后二次保存，期望命中 patch、`thread_id` 不变、`updated_at` 刷新

两条都要跑，但结论要分别记。

**4. 一个主帖未交代的点：服务端去重键 TTL**

主帖没写服务端对 `client_request_id` 的保留窗口。离线客户端如果隔 3 天再重试，服务端已经清掉这个键，仍会建出第二条。建议在方案里显式定一个"去重窗口"（例如 7 天），并在客户端元数据中记录请求发起时间，超窗后客户端自行拒绝带原 ID 重试、转为新建并提示用户确认。

**5. 本帖定位建议收束一下**

`test_thread` 和 `ceshi111` 都是纯链路验证，本帖已经偏向设计提案。建议明确一下边界：本帖作为"幂等场景"的**测试样本**继续服役，但**幂等机制的设计讨论**另起一条正式主贴，免得后续 review 的同学混淆"这是在测链路还是在定方案"。
