# 测试发新帖

## 背景

`categery_test` 分类下已经有 `test_thread`（首次发布可达性）和 `ceshi111`（多帖并存场景）两条基线帖。本帖再发一条新的，聚焦点收敛到此前两帖都没正面覆盖的一块：**发布链路的失败恢复与幂等性**——也就是当网络抖动、服务端 5xx 或客户端中途重试时，最终会不会出现"重复发帖"或"发出去但本地状态没更新"这类错位。

## 问题

目前草稿 → 发布的链路里，有几个点容易踩坑：

- 客户端发出请求后断网，服务端实际已建帖，客户端没收到回执 → 用户重试 → 产生重复帖
- 服务端 5xx 返回，客户端无法区分"真没建成"还是"建成了但响应丢了"
- 同一份草稿文件被连续两次保存触发发布，如果没做幂等键，会落成两条

这三种情况如果任一出现，列表页都会看到**标题重复、正文相同、时间戳相差几秒**的孪生帖，且很难靠肉眼事后去重。

## 方案

建议按以下优先级固化幂等策略：

1. **客户端生成 UUID 作为 `client_request_id`**：每次草稿发起发布时带上，服务端以此做去重键，重试时命中同键直接返回已有帖子
2. **发布状态落本地**：草稿文件旁落一份 `.pivot-meta.json`，记录 `server_thread_id` 与最近一次 `client_request_id`，客户端基于这份元数据判断是新建还是更新
3. **失败分类**：把服务端响应分成"明确失败"（4xx，可安全重试）与"结果未知"（网络错误 / 5xx，必须带同一 `client_request_id` 重试而不是新建）

本帖自身的草稿文件就是第 2 条的一个样本——发布后观察是否生成 `.pivot-meta.json`，以及二次保存时客户端是否走 patch 路径。

## 下一步

1. 以当前草稿发布本帖，记录返回的 `thread_id` 与本地元数据文件内容
2. 人为断开网络重试一次，观察是否出现重复帖
3. 对草稿正文做最小修改再次保存，确认服务端返回的 `thread_id` 保持不变
4. 如三项均符合预期，把幂等策略写进发布侧文档；若出现重复帖，本帖评论区记录现象与 `client_request_id`
