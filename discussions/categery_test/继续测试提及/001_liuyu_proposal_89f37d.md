# 继续测试提及

## 背景

这已经是 `categery_test` 分类下第三条与"提及"相关的新帖，且标题与上一帖完全相同（`继续测试提及`）。前两轮分别覆盖了"首次 @"和"连续两条 @ 同一人"两个场景，本轮在此基础上叠加一个新变量：**同标题多帖**对发布链路、列表展示与通知系统的影响。

## 问题

同标题多帖在多个环节都存在被错误合并或难以区分的风险：

- 列表页若仅以标题作为视觉锚点，三条同名帖子会让用户难以辨认
- 详情页跳转若错误地按标题去重或路由，可能把后发的帖子映射到先发的 URL
- 飞书通知若按"标题 + 接收人"做幂等键，第三条 @ 可能被静默吞掉
- 服务端存储如果对（分类 + 标题）建了唯一约束，本帖应当直接发布失败而不是默默写入

## 验证目标

- 三条帖子在数据库与列表页都以独立 ID 共存，互不覆盖
- 列表页能用时间戳、作者头像或序号等辅助信息区分同名帖
- 飞书侧本轮收到的是**第三条独立通知**，文案与本帖 `comments` 一致
- 详情页 URL 与帖子 ID 一一对应，刷新或直接访问都能稳定回到本帖

## 验证步骤

1. 发布本帖，记录帖子 ID、URL 与时间戳
2. 在分类列表页确认三条同名帖按时间倒序排列，且每条的链接指向各自 ID
3. 分别访问三条详情页，核对正文、`comments`、@ 渲染互不串
4. 在被 @ 用户飞书侧确认本轮收到一条新通知，跳转目标为本帖
5. 若任一环节出现合并、覆盖或通知缺失，立即在本帖下跟帖记录现象与帖子 ID 对照

## 关注点

- **唯一标识应为 ID 而非标题**：业务侧任何"按标题判断"的逻辑都需要纠正为"按 ID 判断"
- **列表页可读性**：同名帖密集出现时，是否需要在 UI 上追加"#2 / #3"之类的辅助序号，待本轮观察后再讨论
- **通知疲劳**：连续三轮 @ 同一人虽然是测试场景，但生产环境也可能出现，相关频控策略需要在另一议题中专门梳理

## 下一步

本帖跑完后，提及链路的"单次 / 连续 / 同标题连续"三种基线场景就齐了。后续计划把测试矩阵扩展到：多用户同时 @、跨分类 @、回复中 @，以及无效 open_id 的容错。本轮如出现异常，先在本帖下定位再推进后续。
