# 测试提及

## 背景

本帖在 `categery_test` 分类下专门用于验证**新帖发布时 @ 提及链路**：草稿 frontmatter 中的 `mentions.open_ids` 是否能在发布后正确解析为帖子内的 @ 用户，并触发对应的飞书通知。

此前的 `test_thread`、`ceshi111`、`测试发新帖` 几条帖子都未携带 `mentions` 字段，覆盖的是"无提及"基线场景。本帖补齐"有提及"路径。

## 验证目标

- 草稿顶部 `mentions.open_ids` 中列出的 open_id 在帖子详情页正确渲染为 @ 链接，可点击跳转到对应用户
- `mentions.comments` 字段作为通知文案被原样推送到被 @ 用户的飞书
- 通知只对 `open_ids` 中实际列出的人触发，不出现误推
- 后续对该草稿文件的迭代编辑，不应在每次保存时重复推送通知（仅首次发布触发）

## 验证步骤

1. 以当前草稿发布本帖，记录帖子 ID 与发布时间戳
2. 在被 @ 用户（刘昱）的飞书侧确认是否收到通知，核对通知正文是否与 `comments` 字段一致
3. 在帖子详情页检查 @ 段落的渲染样式，以及点击 @ 后的跳转目标
4. 对本草稿做一次小幅正文修改后再次触发发布，观察通知是否被去重
5. 与 `test_thread` 详情页对比，确认无 `mentions` 的帖子不出现任何 @ 痕迹

## 关注点

- **open_id 校验**：若 `open_ids` 中混入无效或已离职用户的 ID，应有清晰的失败提示而非静默吞掉
- **comments 必填语义**：`comments` 为空字符串时应在草稿层就拦截，避免推送一条空内容的飞书通知
- **正文与提及解耦**：正文 Markdown 中不再手写 `@刘昱`，全部由 frontmatter 驱动，确保渲染来源单一
- **通知节流**：同一帖子的多次编辑不应放大为多次推送，防止打扰

## 下一步

发布后若 @ 链路全部符合预期，本帖将作为"含 mentions 的新帖"基线样本保留；若出现通知未达、重复推送、open_id 解析失败等任一异常，请直接在本帖下跟帖记录现象、时间戳与被 @ 用户侧的截图，便于定位是草稿解析、发布服务还是飞书推送环节的问题。
